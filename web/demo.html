Today, only humans can pay for APIs — you need a credit card, an email, an account. AIs ("agents") can't do any of that. So agents can't really buy or sell. Proprietor is a tiny shop run by an AI that earns and spends digital dollars (USDC) all by itself — proving agents can actually do commerce.

Explain like I'm 5 the whole loop

Imagine a robot shopkeeper. It has a piggy bank full of digital dollars (USDC). When you ask it a question about a company, it:

  • 1. Tells you the price first.
  • 2. You drop coins in its piggy bank.
  • 3. It pays another robot (a researcher) to do the work.
  • 4. It gives you the answer plus a receipt showing every coin in and out.
  • 5. If a job costs too much, it raises its price all by itself.

No human approves the trades. The piggy bank itself is the rule book.

The problem why agents can't do commerce yet

  • APIs charge by monthly subscription, not per request — bad fit for an agent that makes 4 calls a day.
  • Sign-up needs an email, a card, a billing address — an AI has none.
  • An agent can't spend money it doesn't have authority over; without a wallet it asks a human, and the human becomes the bottleneck.
  • There's no shared way for an agent to discover a price on the wire and decide if the trade is worth it.

Result: the agent economy is stuck. Agents can think but they can't transact.

What we're solving the missing rails

  • Per-call price discovery — 402 Payment Required over HTTP (x402): the server itself quotes the price.
  • Programmable money — Circle Agent Wallet so the AI is the account, not a human under it.
  • Two-sided commerce — the agent earns AND spends, end-to-end, no human in the middle.
  • Receipts on-chain — every job lands a tx hash; books reconcile themselves.
  • Autonomous pricing — the CFO loop reprices when costs rise, so the business stays solvent.

Why this matters for Circle and Circle's customers

For Circle

Circle wants USDC to be the unit of account on the agent web. Proprietor is the canonical proof: a single agent runs a profitable business end-to-end on Circle's stack — Agent Wallet + x402 batched settlement + Arc Testnet + Marketplace inspect. If this works, every API on the internet can become payable by AIs without giving up custody to a card processor.

For buyer-agents

No accounts. No keys. No invoices. Discover a price, pay, get the answer — like dropping a coin in a vending machine. Costs scale with what you actually use, and the price you see is the price you pay.

For sellers / app builders

Charge per request. No chargebacks, no card-processor cut, no subscription billing code. Settlement in USDC, batched and on-chain. Your API becomes addressable by every agent on the network — not just humans willing to enter a credit card.

Treasury concepts how the piggy bank works

  • Treasury — the Circle Agent Wallet the shop owns. It's the company account. Holds all USDC. The agent can sign on its own behalf; no human keys exist outside Circle's custody.
  • Balance — current USDC in the wallet. Read before every quote so the agent never promises work it can't pay for.
  • Runway — balance ÷ recent average loss per job. If runway dips below N jobs, the CFO raises the price before running out.
  • Wholesale cap (max_wholesale_usdc) — hard ceiling per job on what the agent will pay the supplier. Above it, the job is refused outright.
  • Minimum margin (min_margin_usdc) — the agent won't accept a job whose expected profit is below this. Defends the business from being out-priced into the ground.
  • Approval threshold (autonomous_max_usdc) — spend ≤ threshold runs autonomously; above it escalates to a human. Autonomy is the default, supervision is the exception.
  • Decline-before-charge — if the agent can't honour a paid request, the route returns 409. x402 has no refunds, so we never collect a 402 we can't fulfil.
  • Settlement state — settled (final on Arc) or pending-batch (Circle Gateway batching). The ledger records both states transparently.
  • Ledger — append-only SQLite of every job: order ID, decision branch, revenue, wholesale, margin, tx hashes, LLM reasoning. The CFO reads its own ledger to learn.

FAQ tech · practical · cybersecurity

Tech & architecture
What exactly is x402?

An HTTP-native payment scheme. The server replies 402 Payment Required with the price, the chain, and the seller address. The client pays, retries with a payment proof header, and the server fulfils. We use @circle-fin/x402-batching on the server and Circle CLI as the client.

Which chain & why?

Arc Testnet — Circle's EVM-compatible chain optimized for stablecoin settlement. Sub-second finality, USDC-native. We never touch mainnet money in the demo.

Is the agent really autonomous?

Yes within a budget. The CFO is a Claude Agent SDK loop that reads the budget rulebook, inspects the supplier price, and decides allow / deny / reprice. Above an approval threshold it escalates — autonomous by default, supervised only at the tail.

What's the LLM/research stack?

Tavily for web search, Nebius TokenFactory for LLM inference (extract + synthesise), FastAPI for the engine, Express for the storefront, TS + Vitest for the money layer, uv/pytest for the engine tests.

How are receipts verifiable?

Each receipt contains revenue_tx_hash and supplier_tx_hash. Anyone can look them up on Arc Testnet's explorer to confirm the two-sided settlement.

What happens if the engine fails mid-job?

x402 is decline-before-charge: unavailable depths return 409 (not a payable 402). If fulfilment fails after a charge, the receipt records the failure and the agent flags it for the ledger.

Does this scale beyond a demo?

The bottleneck is settlement latency, not protocol. Circle Gateway batches x402 payments so we can pipeline many concurrent buyers. Engine is stateless behind a SQLite cache keyed on (company, depth).

Practical / business
Why would anyone use this instead of Stripe?

Stripe requires a human with a card. This works for agent-to-agent trade where neither side is a human. No accounts, no KYC dance, no minimums, pay-per-call from $0.001 up.

What's the unit economics?

Per job: revenue (e.g. 0.03 USDC) − wholesale paid to supplier (e.g. 0.018 USDC) = margin. The CFO targets a minimum margin; if reality drifts below it the agent reprices on its own.

How does the agent decide the price?

It reads recent ledger margins, the supplier's current quote, and an elasticity heuristic. Above an autonomous threshold it escalates. Pricing logic lives in cfo/repricer.ts and is unit-tested.

Could a buyer-agent get stuck in a fee loop?

No — buyer sets --max-amount on every circle services pay. If the storefront quotes above that, the pay fails fast with no funds moved.

Who pays gas?

On Arc Testnet, payers cover their own gas. Circle Gateway batches settlement to amortise fees across multiple x402 calls. Receipts may briefly say pending-batch.

What's the demo failure mode you're most worried about?

Live external APIs (Tavily/Nebius) flaking on stage. Mitigation: a free /preview path that doesn't move money, and a recorded fallback of the paid run with real on-chain hashes.

Can humans still take over?

Yes. Spend above the approval threshold escalates to a human-in-the-loop step. The budget rulebook is the only thing standing between autonomous and supervised mode.

Cybersecurity
Could the agent be prompt-injected into draining the treasury?

The CFO never executes free-form prompts as payment instructions. Spend is gated by cfo/gate.ts: every payment must pass max-wholesale, min-margin, and per-job caps regardless of what the LLM "decides". The Agent SDK can only call a narrow tool surface — there is no "transfer arbitrary USDC" tool.

What if a supplier overcharges?

Buyer-side: storefront calls supplier with a --max-amount derived from the budget. If the supplier quotes higher, the call fails before any settlement. The decline is logged to the ledger.

What about replay attacks on x402?

x402 payment proofs are bound to a request hash + nonce and verified by the facilitator. The storefront rejects duplicate proofs; the chain rejects duplicate txs.

Where do private keys live?

The Circle Agent Wallet is custodial via Circle — we never see raw private keys. Auth is via Circle CLI session tokens, kept out of the repo (.env is git-ignored).

Is the storefront DOS-resistant?

Free routes (/healthz, /v1/preview) are cheap and rate-limit-friendly. Paid routes are inherently rate-limited by the cost-to-call — flooding 402s costs the attacker USDC. Pay to spam is the natural defence.

What if the LLM hallucinates a profile?

The engine has a validation pass at the deepest tier and a deterministic-fields-by-regex extractor; fuzzy fields carry a confidence flag. The buyer sees both the profile and the cost breakdown to judge value.

Could a malicious buyer submit a poisoned company name?

All input is parsed as a domain string before reaching the engine; the engine sandboxes any URL fetch (timeouts, allow-listed schemes, no follow into private IP ranges).

Are receipts tamper-proof?

The receipt JSON is server-generated and the on-chain tx hashes are independently verifiable. Tampering changes the hash but not the chain — anyone can re-verify.

What about supply-chain attacks on the agent's deps?

Honest answer: this is a hackathon repo — deps are npm install'd at known versions but not yet pinned via lockfile audit. Production hardening would add npm audit signatures + reproducible builds.
